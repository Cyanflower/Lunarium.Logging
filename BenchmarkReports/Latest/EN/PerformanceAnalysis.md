## Caller-Side Throughput: Logger End-to-End

The measurements here reflect the **caller thread** cost of `Log()` — covering `LogEntry` construction and Channel `TryWrite` only, excluding rendering and I/O.

### Single Call (ns)

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| No properties, plain-text message | 188.3 ns | 128 B |
| Single property (params object?[1] allocation) | 199.1 ns | 160 B |
| Three properties (params object?[3] allocation) | 190.4 ns | 200 B |
| Five properties (params object?[5] allocation) | 201.3 ns | 240 B |
| Via ForContext wrapper (LoggerWrapper call) | 185.4 ns | 160 B |

Single-call latency is consistently **185–201 ns** regardless of property count (0–5 properties add less than 16 ns). The allocation comes from the `LogEntry` object (fixed portion) plus the `params object?[]` array (~8 B boxing per property). The `ForContext` wrapper itself adds zero allocation — Context/ContextBytes are precomputed at construction time.

### Batch Amortization

| Scenario | Total Latency | Per-Entry Amortized |
| :--- | :--- | :--- |
| Batch of 100 entries | 20,117 ns | **~201 ns/entry** |

The per-entry amortized cost matches the single-call latency, confirming that Channel writes have no batch amortization effect — each log entry competes independently. The theoretical single-thread throughput ceiling is approximately **5.3 Million calls/sec**; real-world multi-threaded workloads operate well below this, so Channel will not become a bottleneck.

## Allocation Characteristics

| Module | Normal Path Allocated | Notes |
| :--- | :--- | :--- |
| Filter (cache hit) | **0** | Fully zero allocation |
| Parser (cache hit) | **0** | Fully zero allocation |
| LogWriter Text/Color render | **32 B** | Fixed; independent of property count |
| LogWriter JSON render | **64 B** | Fixed; independent of property count |
| Log() caller (no properties) | **128 B** | LogEntry fixed portion |
| Log() caller (per additional property) | **+32 B** | params boxing |
| WriterPool Get + Return | **0** | Pooled path, zero allocation |
| Filter cache miss | 218 B | Rarely triggered |
| Parser cache miss | 1,414 B | Rarely triggered |
| JSON `{@Payload}` fallback path | 656 B | Prefer `IDestructurable`/`IDestructured` instead |

The 128–240 B allocation per `Log()` call is an inherent cost of the current `params object?[]` API, primarily from array allocation and value-type boxing. At an extreme rate of 1 Million calls/sec, this generates approximately 120–240 MB/sec of Gen0 allocation pressure — a level the .NET GC can typically sustain without promoting to Gen1/Gen2.

## Usage Recommendations

### High-Frequency Logging (> 1 Million calls/sec)

- Prefer zero-property or low-property (≤ 3) calls to minimize `params` boxing allocation
- Use `LogEntryChannelTarget` instead of `StringChannelTarget` to avoid rendering and decoding overhead on the channel path
- Implement `IDestructurable` or `IDestructured` for complex object destructuring — avoids the `JsonSerializer.Serialize` fallback path (~592 B extra allocation per call)
- Keep context strings as static constants to prevent Filter cache thrashing

### Conventional Logging (< 100 K calls/sec)

- All paths have ample headroom; no special optimizations are needed
- GC impact of JSON vs Text format is negligible at this frequency

### AOT Scenarios

- Register a `JsonSerializerContext` via `GlobalConfigurator.UseJsonTypeInfoResolver()`; otherwise `{@Object}` destructuring falls back to `ToString()`
- Types implementing `IDestructurable`/`IDestructured` are not subject to AOT restrictions

## Hot Path: Filter

Filtering runs on the background thread for every dispatched log entry and is one of the highest-frequency operations.

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| No rules, level passes (cache hit) | 8.663 ns | 0 |
| Include rule, context matches — pass (cache hit) | 8.889 ns | 0 |
| Include rule, context mismatch — reject (cache hit) | 8.921 ns | 0 |
| Exclude rule, context not in list — pass (cache hit) | 8.359 ns | 0 |
| Exclude rule, context matches list — reject (cache hit) | 8.734 ns | 0 |
| Cache miss (triggered after 2,048 unique contexts exceeded) | 237 ns | 218 B |

The **cache-hit path** (normal operation) is a constant **~8–9 ns with zero allocation**, regardless of rule type or pass/reject outcome. This indicates that the filter branches are consistently predicted and remain resident in L1 cache.

**Cache misses** (237 ns) only occur when the 2,048-entry context cache is exhausted and rebuilt. For the standard usage pattern where context is a static string (class name or module name), this path is practically never reached.

> **Note**: If you regularly encounter the 2,048-entry cache limit, please consider opening an Issue to help the project better understand real-world usage patterns and evaluate an appropriate cache size.

## Hot Path: LogParser

Template parsing also runs on the background thread, with parse results cached for reuse.

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| Plain text, no placeholders (cache hit) | 13.65 ns | 0 |
| Single-property template (cache hit) | 11.01 ns | 0 |

| Template with escape `{{ }}` (cache hit) | 13.94 ns | 0 |

The **cache-hit path** is a constant **11–18 ns with zero allocation**. Template complexity (property count, alignment, format specifiers, destructure prefix) has minimal effect on the cache-hit cost.

**Cache misses** (1,281 ns) occur on first parse or after cache overflow. In a typical service, template strings form a finite static set, so the cache hit rate approaches 100%.

## Rendering Performance: LogWriter

Rendering runs on the background thread and reuses Writer instances via `WriterPool`. **Fixed allocation per render is just 32 B (Text/Color) or 64 B (JSON)**, regardless of property count.

### Text Writer (ns)

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| Plain-text message (no properties) | 328.91 ns | 32 B |
| Single property | 356.61 ns | 32 B |
| Four properties | 446.02 ns | 32 B |
| Alignment + formatting (`{Count,8:D} {Percent:P1}`) | 526.32 ns | 32 B |
| Numeric/Formattable (int, DateTimeOffset, TimeSpan) | 465.06 ns | 32 B |

### Color Writer (ns)

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| Single property (with ANSI color escapes) | 438.36 ns | 32 B |
| Four properties | 605.66 ns | 32 B |
| Alignment + formatting | 626.05 ns | 32 B |
| Numeric/Formattable | 591.34 ns | 32 B |

Color Writer adds approximately **100–170 ns** over Text Writer, coming from the ANSI escape byte sequences written around each field. Allocation is identical.

### JSON Writer (ns)

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| Single property (includes RenderedMessage + Properties fields) | 506.76 ns | 64 B |
| Four properties | 759.57 ns | 64 B |
| Numeric/Formattable | 772.94 ns | 64 B |
| `{@Payload}` IDestructured (pre-serialized bytes, zero-allocation path) | 634.73 ns | 64 B |
| `{@Payload}` anonymous type (JsonSerializer.Serialize fallback) | 979.06 ns | **656 B** |

JSON Writer maintains a fixed **64 B** allocation regardless of property count. The `IDestructured` interface (pre-serialized bytes) is the recommended path for object destructuring — **35% faster** and **90% less allocation** than the `JsonSerializer.Serialize` fallback. For custom types that require destructuring, implementing `IDestructurable` or `IDestructured` preserves the fixed-allocation characteristic.

### Object Pool

| Scenario | Latency | Allocated |
| :--- | :--- | :--- |
| `WriterPool.Get<LogTextWriter>()` + `Return()` (pooled path) | 77.23 ns | 0 |
| `new LogTextWriter()` (direct allocation, for comparison only) | 54.56 ns | 520 B |

The pooled path (77 ns, 0 allocation) is marginally slower than direct allocation (54 ns, 520 B), but eliminates GC pressure at high frequency. WriterPool holds up to 128 instances; Writers whose internal buffer exceeds 32 KB are not returned to the pool (the underlying array is released instead).

## ChannelTarget

The `ChannelTarget` family delivers log entries asynchronously to external consumers (network, custom processors, etc.).

| Variant | Latency | Allocated | Suitable For |
| :--- | :--- | :--- | :--- |
| `LogEntryChannelTarget` (zero encoding, passes reference) | 50.30 ns | 0 | Consumer needs the full LogEntry object |
| `ByteChannelTarget` (render + byte[] copy) | 992.23 ns | 136 B | Consumer writes directly to a network or file byte stream |
| `StringChannelTarget` (render + UTF-8→string decode) | 1,065.78 ns | 208 B | Consumer requires a string type |

The 50 ns of `LogEntryChannelTarget` covers only the Channel write; rendering is handled by the consumer. The ~1 μs of `ByteChannelTarget` and `StringChannelTarget` includes a full render pass — the overhead beyond a standalone LogWriter render (~330 ns) comes from the Channel write and the byte copy or decode step. If the consumer handles raw bytes, `ByteChannelTarget` saves ~74 ns and 72 B over `StringChannelTarget`. This design offloads the most time-consuming rendering work (~300–800 ns) entirely from the caller thread to the background consumer thread, maximizing throughput on the business-logic path.
