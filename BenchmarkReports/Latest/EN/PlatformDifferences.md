### I. Filter — Cache Hit / Cache Miss

#### Cache Hit (ns)

| Platform | No Rules | Include Pass | Include Reject | Exclude Pass | Exclude Reject |
| :--- | :--- | :--- | :--- | :--- | :--- |
| LinuxLaptop | 8.7 | 8.9 | 8.9 | 8.4 | 8.7 |
| LinuxServer | 9.8 | 9.7 | 9.5 | 9.3 | 9.5 |
| WindowsPC | 9.4 | 9.5 | 10.3 | 9.6 | 9.3 |
| WindowsServer | 9.1 | 9.0 | 9.0 | 8.9 | 9.1 |

LinuxLaptop has the fastest cache hits (~0.5–1 ns advantage) due to the i7-8750H's low single-core L1/L2 latency — `ConcurrentDictionary` lookups are highly sensitive to cache latency. WindowsPC's Include Reject (10.3 ns) is the slowest across all platforms; the likely cause is slightly weaker branch prediction in the Windows 11 RyuJIT codegen.

#### Cache Miss

| Platform | Latency (ns) | Allocated |
| :--- | :--- | :--- |
| LinuxLaptop | 237 | 218 B |
| LinuxServer | 167 | 192 B |
| WindowsPC | 204 | 192 B |
| WindowsServer | 150 | 192 B |

**Key Difference 1 — Allocation size:**
LinuxLaptop allocates 218 B, 26 B more than the other three. It is the only platform running .NET **10.0.2** (the rest use 10.0.4/5). The extra bytes are likely caused by an internal data-structure layout change between minor runtime versions (e.g., `ArraySegment` or `ConcurrentDictionary` node alignment adjustment).

**Key Difference 2 — Miss latency:**
The two EPYC machines (167/150 ns) outperform the two Intel machines (237/204 ns). EPYC's higher memory bandwidth benefits cache-rebuild operations after eviction (new dictionary allocation, HashCode computation) through faster L3 throughput.

**Key Difference — LinuxServer JSON performance is notably worse:**

`Utf8JsonWriter` contains branch-dense string lookup operations. The same EPYC on Windows Server achieves 810 ns for JSON 4 properties vs 923 ns on Linux, ruling out the CPU itself; **OS-level differences** (heavier memory barrier or syscall paths for `Utf8JsonWriter` on Linux) are the primary cause.

### IV. LoggerThroughput — Caller-Side Throughput

This is the benchmark with the largest cross-platform divergence.

#### Single Log() Call (ns)

| Platform | No Props | 1 Prop | 3 Props | 5 Props | ForContext |
| :--- | :--- | :--- | :--- | :--- | :--- |
| WindowsPC | 153 | 168 | 169 | 185 | 163 |
| LinuxLaptop | 188 | 199 | 190 | 201 | 185 |
| WindowsServer | 380 | 250 | 321 | 303 | 397 |
| LinuxServer | 576 | 433 | 539 | 539 | 692 |

**Key Difference 1 — WindowsPC is the fastest:**
The i7-7700's high clock speed (3.6 GHz) minimises the latency of Channel `TryWrite` lock/lock-free operations, placing it ~20% ahead of LinuxLaptop and 2–4× ahead of the EPYC machines.

**Key Difference 2 — LinuxServer is dramatically slower and highly unstable:**
- No-props: 576 ±133 ns (CV **23%**); all other platforms have CV in the 3–10% range
- ForContext: 692 ±145 ns (CV 21%)
- Same CPU as WindowsServer (EPYC 7542), yet **1.5–2× slower** on Linux with severe jitter

This is the most noteworthy cross-platform finding. The root cause lies in .NET Channel's `TryWrite` mechanism, which uses `Interlocked.Exchange` + `ManualResetValueTaskSourceCore` for wake-up signaling. On Linux this maps to the `futex` syscall, and EPYC's NUMA topology combined with the Linux scheduler introduces larger memory-coherency latency at 8 cores. The Windows kernel applies more aggressive optimizations to `Interlocked` operations.

#### Batch of 100 Entries (ns, total per batch)

| Platform | Batch Latency | Per-Entry Amortized | vs Single Call |
| :--- | :--- | :--- | :--- |
| WindowsPC | 17,222 | 172 ns/entry | ≈ single |
| LinuxLaptop | 20,117 | 201 ns/entry | ≈ single |
| WindowsServer | 27,812 | 278 ns/entry | ≈ single |
| LinuxServer | 62,821 | 628 ns/entry | ≈ single |

The per-entry amortized cost aligns closely with the single-call latency on all platforms. This confirms that Channel has no meaningful batching amortization effect — every log write competes independently.
